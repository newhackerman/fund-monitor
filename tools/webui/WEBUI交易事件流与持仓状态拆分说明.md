# Web UI 交易事件流与持仓状态拆分说明

## 为什么要拆
之前 Web UI 的“最新交易”区域同时展示了：
- `BUY / SELL`（事件类型）
- `OPEN / CLOSED`（持仓状态）

这两个维度不是一回事：
- `BUY / SELL` 回答的是：**发生了什么事件**
- `OPEN / CLOSED` 回答的是：**这笔仓位现在处于什么状态**

如果把它们混在一层展示，用户会很容易误读成：
- `BUY = 持仓中`
- `SELL = 已结束`
- 或者把 `BUY + CLOSED` 这种记录看成异常数据

但从 `trades.json` 的契约看，这其实是允许存在的：
- `BUY + OPEN`
- `BUY + CLOSED`
- `SELL + CLOSED`

所以前端必须把“事件”和“状态”拆成两张不同口径的视图。

---

## 现在怎么展示

### 1. 交易事件流
用途：看事件发生了什么。

展示来源：
- `trades.json`

展示重点：
- 时间
- 代码
- 事件类型（只看 `BUY / SELL`）
- 价格
- 收益%
- 持仓状态（作为附属字段展示，不再与事件混成一个筛选口径）
- 原因

理解方式：
- 这里的主维度是 **事件流**
- `BUY` / `SELL` 是第一层语义
- `OPEN` / `CLOSED` 只是补充说明这条交易记录当前关联的状态

---

### 2. 持仓状态视图
用途：看当前还有哪些仓位没平。

展示来源：
- `trades.json` 中的 `BUY + OPEN`

展示重点：
- 开仓时间
- 代码 / 名称
- 买入价
- 最新价
- 浮盈亏%
- 持有分钟
- 状态
- 原因

理解方式：
- 这里的主维度是 **当前持仓**
- 只回答“现在仓位是什么情况”
- 不再混入 SELL 事件

---

## 这次改动的意义

### 对用户认知更清晰
用户可以很直观地区分：
- 我今天/最近发生了哪些交易事件
- 我现在手上还有哪些持仓

### 对前后端口径更一致
与 `运行数据字段契约说明.md` 保持一致：
- `signals.json` 更适合看信号事件流
- `trades.json` 更适合看交易闭环和仓位状态
- `BUY/SELL` 与 `OPEN/CLOSED` 不再混成一个解释层

### 对后续扩展更安全
以后如果增加：
- 部分止盈
- 分批卖出
- 同一 code 多笔独立持仓
- 状态机更复杂的交易记录

这个拆分结构也更容易继续扩展，而不会把 UI 语义越做越乱。

---

## 当前页面对应关系

### 最新信号事件流
- 数据源：`signals.json`
- 语义：策略信号层

### 交易事件流
- 数据源：`trades.json`
- 语义：交易事件层

### 持仓状态视图
- 数据源：`trades.json` 中 `BUY + OPEN`
- 语义：当前仓位层

---

## 一句话总结
这次拆分的核心不是“多加一张表”，而是把：
- **事件发生了什么**
- **仓位当前是什么状态**

从根上拆成两个清晰维度，避免 Web UI 继续把 `BUY/SELL` 和 `OPEN/CLOSED` 混成一团。
